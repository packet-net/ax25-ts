## [Unreleased]

### Fixed

- **figc4.5 Timer-Recovery stored-frame drain under-advanced V(R) — simultaneous-bidirectional SREJ re-delivery / non-convergence (TS↔C# parity)** — mirrors the C# reference fix ([packet.net#286](https://github.com/M0LTE/packet.net/pull/286)). figc4.5's (Timer Recovery) in-sequence `I_received` stored-frame **drain loop** draws its body as `V(r) := V(r) - 1`, where the structurally-identical figc4.4 (Connected) drain — same path, same pre-loop `V(r) := V(r) + 1` — uses `V(r) := V(r) + 1`. The drain delivers each consecutively-stored (SREJ-gap-filled) frame and must *advance* V(R) past it; the decrement cancels the pre-loop increment the moment one stored frame is drained, so a station recovering an SREJ gap *while in Timer Recovery* delivers the gap-filled frames but leaves V(R) pointing at an already-delivered sequence number — and the peer's next genuine (still-unacknowledged-window) retransmit is then taken for new data and re-delivered, and the link fails to converge. This is a confirmed figure defect ([packethacking/ax25spec#47](https://github.com/packethacking/ax25spec/issues/47), confirmed by the spec author; figc4.4 being already correct is the proof the figc4.5 `-` should be `+`; corroborated by direwolf's `dl_data_indication` drain, which advances `state->vr` and never decrements). Fixed by a new **default-on** session quirk `ax25Spec47TimerRecoveryDrainAdvancesVR` (off under `strictlyFaithfulSessionQuirks`) that, in `ActionDispatcher`, rewrites the verb `V(r) := V(r) - 1` → `V(r) := V(r) + 1`. The verb appears in **exactly three** transitions — all the figc4.5 drain loops (verified against the `ax25sdl` tables: 3 occurrences, all in `timer_recovery.g`), nowhere else — so the rewrite is precisely scoped, no trigger gate needed, inert everywhere else; the SDL tables stay faithful (the decrement is what `strictlyFaithful` runs). This single change closed **both** residual bidirectional-SREJ edges the #285 mirror left open: the shared low-n case (mod-8 SREJ k=4, A submits one payload while B submits two, `LCG(1)` budget 3 → A delivered B's stream twice as `[0x80,0x81,0x80,0x81]`) and the simultaneous-bidirectional ring-wrap seeds (mod-8 `LCG(1828421821)`, mod-128 `LCG(4203678057)`, k+2 simultaneous) that previously tipped TS into the figc4.5 go-back-N path where the C# reference stays in Connected selective replay — the under-advanced V(R) was what tipped it, so the correct advance keeps TS in selective replay too (no separate sequencing change needed, runtime deferring to the C# reference). The known-failure file `tests/properties/srej-recovery-duplicate-delivery.known-failure.test.ts` is removed and its cases folded into the loss-recovery property tier as passing assertions: the R1 minimal repro, a quirk-isolation tripwire (with *only* `ax25Spec47…` off the `[0x80,0x81,0x80,0x81]` defect reproduces exactly and the link does not converge — proving the decrement is the sole cause), a 400-case simultaneous-bidirectional low-n SREJ sweep (both modulos, REJ + SREJ), and the two ring-wrap reference pins; the existing simultaneous-bidirectional wrapping-burst property now covers SREJ as well as REJ (verified 0 failures over a dense 50 000-case SREJ ring-wrap sweep). New dispatcher-level quirk-isolation tests in `tests/RecoveryRuntimeQuirks.test.ts` (on → advance, off → decrement, inert on `+1`). The transition-coverage ledger holds at 238/250 (the rewrite changes an action verb inside three existing transitions, not the transition set). Delete the quirk once `ax25sdl` ships a corrected figc4.5.
- **mod-8 SREJ sequence-ring-wrap duplicate delivery (TS↔C# parity)** — stop the SREJ selective-replay path replaying acknowledged I-frames, mirroring the C# reference fix ([packet.net#285](https://github.com/M0LTE/packet.net/pull/285)). Under mod-8 + SREJ, a bulk transfer of ≥ 8 I-frames in flight (so V(S) wraps the 0–7 ring) plus a single drop made the sender replay stale frames that the receiver — whose forward window at k = 7 mod-8 covers 7 of 8 sequence numbers — could no longer disambiguate from new data once its V(R) had wrapped, so it re-delivered them and the link desynced (the `[1..8, 1, 2]` shape). Three changes on the recovery path (`src/sdl/action-dispatcher.ts` + `session-context.ts`): the selective-replay verb (`pushOldIFrameNrOnQueue` → `emitOldIFrame(..., selectiveReplay=true)`) now (1) refuses to replay a non-outstanding N(S) via `isOutstanding` (the live send window `[V(a), V(s))`) and (2) replays each N(S) at most once per recovery cycle via `selectivelyRetransmittedSinceAck` (cleared on V(a) advance, dropped per-N(S) on a fresh emit); and (3) `sentIFrames` is pruned of acked entries on every V(a) advance (`pruneAcknowledgedSentIFrames`). The go-back-N `Invoke_Retransmission` path stays unguarded (genuine loss must still resend the window via T1). Recovery now holds n ≤ 7 AND n ≥ 8, mod-8 AND mod-128, REJ AND SREJ; unidirectional ring-wrap regression and bidirectional go-back-N wrapping bursts are asserted green in the loss-recovery property tier. (A rare residual under SIMULTANEOUS bidirectional SREJ — a separate recovery edge, partly shared with the C# reference — was pinned here at the time; it is now closed by the figc4.5 Timer-Recovery drain fix above ([packet.net#286](https://github.com/M0LTE/packet.net/pull/286)) and folded into the loss-recovery property tier.)
- **`canTransmitIFrame` state gate (TS↔C# parity)** — `SdlSessionDriver.canTransmitIFrame()` now returns false outside `Connected` / `TimerRecovery`, mirroring C# `Ax25Session.CanTransmitIFrame` ([packet.net#263](https://github.com/M0LTE/packet.net/pull/263)). Data submitted while the session is still establishing (`AwaitingConnection` / `AwaitingV22Connection` — figc4.3 t09 buffer-while-connecting) now buffers on the I-frame queue and drains once Connected, instead of throwing: without the gate the post-dispatch drain popped the just-buffered frame and routed `I_frame_pops_off_queue` into the `push_frame_on_queue` verb, which has no `DL_DATA_request` trigger to read. This was the pre-existing parity divergence flagged by the V5a coverage battery; its figc4.3 t09 sub-step is restored now that the path works.

### Added

- **Receive-path frame classifier (`classifyFrame`) — spec-violating frames now raise the matching DL-ERROR instead of being silently processed** (finding #3 of the test campaign; the TS analogue of C#'s `Ax25FrameClassifier`, runtime behaviour deferring to that reference). Before this, the TS receive path mapped a decoded frame to its SDL event with a small `mapKindToEvent` switch that had no notion of a malformed frame: an info-bearing DM (or any info-bearing S / no-info U frame) decoded to a plain `DM`/`RR`/… with a stray info field and was processed as if valid, and an unknown U-frame control byte (plus, latently, every inbound SREJ) fell through to `null` and was dropped — diverging from the C# runtime, which classifies these into the figc4.x error events. New `src/sdl/frame-classifier.ts` mirrors `Ax25FrameClassifier.Classify` octet-for-octet: an information field on an S frame or a no-info U frame (SABM/SABME/DISC/UA/DM) → `info_not_permitted_in_frame` (DL-ERROR M); an unknown U-frame control byte → `control_field_error` (DL-ERROR L); SREJ now maps to `SREJ_received`; FRMR/XID/TEST/UI (which legitimately carry info) and all well-formed I/S/U frames are unaffected. Both receive paths route through it: `Ax25Listener.dispatchInbound` (cached **and** cache-miss branches) and `Ax25Session._handleFrame` (the `Ax25Stack` facade), replacing both copies of `mapKindToEvent` — so a malformed info-bearing frame on an established link now fires DL-ERROR + `Establish_Data_Link` (re-establish: SABM mod-8 / SABME mod-128 → AwaitingConnection / AwaitingV22Connection) rather than being silently accepted. The SDL dispatch of these error events was already correct (the conformance battery drives them via `inject`); the gap was purely this upstream classifier layer. **Named-flag discipline (CLAUDE rule):** strictness is gated behind a new `Ax25ParseOptions` (`STRICT_PARSE` / `LENIENT_PARSE`) on `decodeFrame`, mirroring the C# `Ax25ParseOptions` record — `allowInfoOnSupervisoryFrames` defaults **on** (`LENIENT_PARSE`, matching the C# parameterless-decoder default the receive path uses), so a trailing info field on an S / no-info U frame is *captured* and surfaced as the data-link error by `classifyFrame` rather than rejected at the wire; `STRICT_PARSE` throws on it at decode instead (matching the C# strict `TryParse` returning false), while FRMR/XID/TEST keep their legitimate info under both. **Tests (mirror packet.net's `Ax25FrameClassifierTests`):** new `tests/frame-classifier.test.ts` (32 — full I/S/U classification incl. the SREJ regression, info-not-permitted on S + each no-info U frame, info-bearing-U-unaffected, unknown-control → `control_field_error`, error-events-carry-no-frame, and the strict-rejects/lenient-accepts paired decode assertions); new `tests/Ax25ListenerErrorPath.test.ts` (4 — an info-bearing DM and S-frame on an established session raise DL-ERROR (M) and re-establish with a SABM on the wire, an unknown U control byte raises DL-ERROR (L) and re-establishes, and a well-formed I-frame trips neither); and a paired `Ax25Stack`-facade error-path case in `tests/session.test.ts`. `classifyFrame`, `Ax25ParseOptions`, `STRICT_PARSE`, `LENIENT_PARSE` are exported from the package index. 786 tests pass (38 files, was 749).

- **Dire Wolf segment inner-PID quirk (`segmentFirstCarriesL3Pid`)** — the TS parity leg of the de-facto-interop segmentation quirk landed C#-side in [packet.net#279](https://github.com/M0LTE/packet.net/pull/279). A new **default-on** session quirk makes the §6.6 segmenter emit *and* accept Dire Wolf's de-facto segment format (which preserves the original Layer-3 PID across segmentation) instead of the figure-literal Figure-6.2 format (which loses it). AX.25 v2.2 Figure 6.2 draws a segment's info field as one `FXXXXXXX` F/X octet followed directly by data — with **no field carrying the original L3 PID** — so a figure-literal reassembly must deliver `PID_NO_LAYER_3` (0xF0) and the PID is lost across the series. The §6.6 prose ("a two-octet header") is ambiguous, and **Dire Wolf (WB2OSZ) — the only known v2.2 segmenter** — reads it to put the original PID as an **inner-PID octet** on the *first* segment: first segment `[F/X][inner-PID][≤ N1−2 data]`, subsequent `[F/X][≤ N1−1 data]`; the inner octet counts toward the budget (segment count = `ceil((len + 1) / (N1 − 1))`, Dire Wolf's `DIVROUNDUP(len + 1, N1 − 1)`), and its reassembler reads the PID back off the first segment. This is a **de-facto-interop quirk, not a figure defect** — it does *not* take the `ax25Spec<NN>` prefix and no `ax25spec` issue is filed (the underlying Figure-6.2 / §6.6 gap is a candidate ax25spec clarification, deliberately not filed). The quirk defaults **on** (interoperate with Dire Wolf out of the box, and the inner-PID octet **fixes the documented `PID_NO_LAYER_3`-on-reassembly limitation** — the original L3 PID now survives a segmented series) and turns **off** under `strictlyFaithfulSessionQuirks`, which keeps the figure-literal `[F/X][data]` / `PID_NO_LAYER_3` form for strict conformance study. **What changed:** `Ax25SessionQuirks` gained `segmentFirstCarriesL3Pid` (on in `defaultSessionQuirks`, off in `strictlyFaithfulSessionQuirks`); `segment()` (in `src/sdl/segmenter.ts`) takes an optional `innerPid` and emits Dire Wolf's format when supplied (figure-literal when `undefined`, requiring N1 ≥ 3 for the inner-PID format so the first segment carries the F/X octet + the inner-PID octet + ≥ 1 data byte — it throws below that, and one byte earlier on the max-segments overflow because the inner octet steals a slot); `Reassembler` gained an `expectInnerPid` constructor flag + a `lastRecoveredPid` accessor (reads the inner PID off the first segment and surfaces it on completion); `SegmentationLayer` (in `src/sdl/segmentation-layer.ts`) reads the quirk **lazily** (`Ax25Listener` builds the shim before its `configureSession` hook sets `ctx.quirks`, so an eager read would miss a strictly-faithful override — the reassembler is now constructed on first receive), passes the L3 PID through to `segment()` on send when the quirk is on, and delivers the reassembled payload with the **recovered original L3 PID** (or `PID_NO_LAYER_3` when off). The `SegmentationLayer.reassembledPid` constant is **renamed `figureLiteralReassembledPid`** (it is now only the strict path's value — matching C#'s `ReassembledPid` → `FigureLiteralReassembledPid` rename). **Tests (mirror packet.net#279):** `tests/segmenter.test.ts` gained inner-PID round-trip + L3-PID recovery coverage, the Dire-Wolf-worked-example byte-pin (N1=4, `"ABCDEF"`, PID 0xF0 → `82 F0 41 42` / `01 43 44 45` / `00 46`), and the inner-PID capacity / minimum-N1 throws, alongside the retained figure-literal cases (now also asserting `lastRecoveredPid` is null); `tests/segmentation-layer.test.ts` flipped the PID test — the default now **preserves the original L3 PID**, with a new `strictlyFaithful` case pinning the figure-literal `PID_NO_LAYER_3` (and that the first-segment second octet is payload, not an inner PID); `tests/Ax25ListenerSegmentation.test.ts` asserts the listener's send seam emits the inner-PID octet by default and the figure-literal form under `strictlyFaithful`; `tests/conformance/segmentation-integration.test.ts` gained default-preserves-L3-PID and strictlyFaithful-figure-literal wired round-trips (the two-station harness gained a `deliveredPids` accessor + a PID-carrying `submitLarge` overload). No interop-stack work in this repo (the Dire Wolf docker round-trip lives in packet.net's `DirewolfMod128Interop`; ax25-ts's interop leg is the separate mod-128 parity row driven by `interop.yml` against the published artefact). 716 tests pass (30 files, was 700).

The battery routes around it (the figc4.3 t09 sub-step is omitted); the fix belongs in a separate, non-test change. 700 tests pass (29 → 30 files).

- **SREJ(mod-128) + segmentation integration** — the TS parity leg of the v2.2 arc V4 (mirrors packet.net#272 + packet.net#273), the last V4 leg. Two parts. **(V4a) SREJ + REJ recovery in the mod-128 sequence space — no recovery-path code change was needed.** The recovery logic in `src/sdl/session-bindings.ts` was already mod-aware: every sequence computation reads `modulus(context)` (8 or 128) and is done `% m` (the `V_s_eq_V_a_plus_k` send-window check, the `ax25Spec40` out-of-window discard guard, the `ns_eq_vr` / `nr_in_window` / `ns_gt_vr_plus_1` comparisons), and v2.2 arc V1 made the frame's `getNs` / `getNr` return 7-bit values on extended frames — so SREJ/REJ recovery works at modulo-128 with the same code that runs mod-8 (the same finding as packet.net#272). Proven + regression-guarded with new `tests/conformance/mod128-loss-recovery.test.ts` (single-frame REJ + SREJ recovery, a multi-frame SREJ burst exercising all three figc4.x SREJ quirks `ax25Spec40/41/42`, and a window-wrap case that seeds V(S)=V(A)=V(R)=124 and transfers eight frames across the 127→0 ring boundary with a drop straddling the wrap), and by extending the generative `tests/conformance/loss-recovery.test.ts` with an `extended` axis so the single-drop and bidirectional-burst sweeps now fuzz *both* modulos (212 → 416 cases). The value here is the proof + the guard, not new code. **(V4b) segmentation integration** — a new `SegmentationLayer` shim (`src/sdl/segmentation-layer.ts`) sits at the DL-primitive boundary, between Layer 3 and a session, exactly as the spec models it (§2.4 / §6.6 / Figure 6.2); the data-link SDL + session stay **unchanged** — segments travel as ordinary I-frames carrying PID `PID_SEGMENTED` (0x08, new in `frame.ts`), so the state machine just sends and receives them. The shim wraps a new standalone `Segmenter` / `Reassembler` (`src/sdl/segmenter.ts`) — created here (ax25-ts had none), mirroring the C# `Segmenter.cs` **as fixed in packet.net#273**: the segment-control byte is `FXXXXXXX` (bit 7 = First, **bits 6:0 = the 7-bit remaining-count**, `0x7F` mask, **max 128 segments** per Figure 6.2, NOT 6-bit/64; de-facto-corroborated by direwolf masking the count with `0x7f`). *Send* (`buildSendRequests`): if the segmenter is negotiated (the `segmenterReassemblerEnabled` flag V3's XID negotiation sets) and the payload exceeds N1, segment it into PID-0x08 `DL_DATA_request`s; otherwise pass one request through unchanged; an over-N1 payload with the segmenter *not* negotiated is **rejected cleanly** (throws — never truncated or sent oversize). *Receive* (`onDataIndication`): a 0x08-PID DL-DATA indication is fed to a per-session reassembler and only surfaced (as one reassembled indication) on the last segment, everything else passing through transparently. **Seam / placement:** the shim is a dedicated, reusable, independently-testable class wired by `Ax25Listener` (and the test harness), *not* buried in the dispatcher/SDL. On receive, the listener's `emitUpward` fan-out (the natural DL-indication boundary) runs each indication through the reassembler before forwarding; the dispatcher's `DL_DATA_indication => emitUpward(...)` is left untouched. On send, a new public `Ax25Listener.sendData(session, data, pid)` routes through the per-session shim (callers wanting a raw never-segmented request can still post a `DL_DATA_request` / `write` directly). **Inner PID on reassembly:** Figure 6.2 carries no field for the original Layer-3 PID through a segmented series, so reassembled data is delivered with `PID_NO_LAYER_3` (0xF0), exposed as `SegmentationLayer.reassembledPid` and documented on the type (mirrors C#'s `ReassembledPid`). **Exit criterion (met):** `tests/conformance/segmentation-integration.test.ts` — a >N1 payload segments on send, reassembles intact on receive over a mod-128 link, and SREJ selectively recovers a dropped segment (asserts an SREJ on the wire, not a T1-timeout go-back-N); plus the mod-8 round-trip, the REJ-recovers variant, and the no-segmenter-oversize reject. The two-station harness gained `segmenter` / `n1` build knobs, a per-endpoint `SegmentationLayer` wired into its `emitUpward`, and a `submitLarge` scenario method (records the whole pre-segmentation payload as one logical submission, so the convergence oracle compares one submission to one reassembled delivery). New `tests/segmenter.test.ts` (16: round-trips incl. the 128-segment 32 640-byte boundary, the 129-segment overflow-throws, header layout, reassembler rejects), `tests/segmentation-layer.test.ts` (8: the shim unit — segment / pass-through / reject, reassembly, the PidNoLayer3 contract), and `tests/Ax25ListenerSegmentation.test.ts` (4: the listener `sendData` seam). The new `SegmentationLayer` / `Segmenter` / `Reassembler` / `PID_SEGMENTED` are exported from the package index. 699 tests pass (24 files / 457 → 29 / 699).

- **MDL XID parameter-negotiation runtime** — the TS parity leg of the v2.2 arc V3 part 2 (mirrors packet.net#271), closing V3 across both runtimes. A new `Ax25ManagementDataLink` driver (`src/sdl/management-data-link.ts`) runs the management data-link FSM — figc5.1 `Ready` / figc5.2 `Negotiating`, from the new `management_data_link` machine in ax25sdl 0.9.0 (exported as `ManagementDataLinkReady` / `ManagementDataLinkNegotiating`) — through the *same* `SdlSessionDriver` / `ActionDispatcher` / `GuardEvaluator` machinery the data-link uses (a proportionate 2-state reuse, not a second interpreter): the driver is now parameterised by an optional `statePages` map + MDL dispatcher hooks + extra guard bindings, defaulting to the data-link behaviour so existing sessions are unaffected. The MDL machine runs against its own isolated `Ax25SessionContext` (its `RC` / `NM201` / `TM201` never disturb the live data-link `RC` / T1/T2/T3 — `NM201` maps onto the MDL context's `n2`, `TM201` is a new distinct timer name added to `TimerName` so a shared scheduler never collides), and applies the *negotiated* parameters to the **real** data-link context — turning SREJ / segmentation / modulo / window / T1 / N2 from config-flag-forced defaults into negotiated link parameters. New MDL dispatcher hooks (`sendXidCommand` / `applyNegotiatedParameters` / `sendMdl` / `setVersion20` / `onMdlNegotiateRequest`) ride on the `ActionDispatcher` (no-op / data-link defaults otherwise), and the new MDL verbs get dispatch arms (`XID_command`, `Start`/`Stop TM201`, `Apply Negotiated Parameters`, `MDL_NEGOTIATE_confirm`, `MDL_ERROR_indicate_B/C/D`); the shared `set_version_2_0` verb now routes through the `setVersion20` hook (data-link default clears `isExtended` only; the MDL override installs the full §1436 set). `src/sdl/xid-negotiator.ts` implements the §6.3.2 "reverts-to" merge (`applyNegotiated`) per parameter — HDLC Optional Functions (reject scheme + modulo) → **lesser** of the two offers so SREJ/mod-128 survive only if both sides offer them (§6.3.2 ¶1426); Classes of Procedures (duplex) → lesser (¶1424); Window k + N1 → notification/**min** (¶1428/¶1430); Ack Timer T1 + Retries N2 → **greater** (¶1432/¶1434); the segmenter/reassembler bit → mutual-capability AND (¶1419); a field absent from both offers retains the context's current value (¶1024) — plus `applyVersion20Defaults`, the full §1436 version-2.0 default set installed on a FRMR-of-XID (half-duplex, implicit reject, **mod-8**, N1=256 octets, **k=7** — the §1436 fallback value, NOT the mod-8 XID offer default of 4 — T1=3000 ms, N2=10; not merely clearing the extended flag). Integration: the data-link figc4.6 UA-received path raises `MDL-NEGOTIATE Request` after a successful v2.2 connect, which `Ax25Listener` (and the two-station harness) routes to the session's MDL driver to open the XID exchange (send XID command, start TM201, → Negotiating); inbound XID / FRMR-of-XID frames are routed to the MDL while it negotiates, disambiguating XID **command** vs **response** by the C/R bit — command → responder (the un-transcribed figc5.1 `respondToXidCommand` helper: merge per §6.3.2, reply with an XID response F=1 carrying the agreed values), response while negotiating → apply, FRMR while negotiating → v2.0 fallback. New `xid()` / `frmr()` frame factories + `"XID"` / `"FRMR"` classification (U-frame control 0xAF / 0x87, info-bearing, no PID); a new `segmenterReassemblerEnabled` context flag. The MDL pages are a prose-derived bootstrap (hand-authored upstream from §6.3.2 + App-C5.3 prose; figc5.x never redrawn, marked `verification_pending`); the figc5.3–figc5.8 per-parameter reverts-to subroutines are collapsed to a single `Apply Negotiated Parameters` placeholder whose runtime body `src/sdl/xid-negotiator.ts` supplies, and the figc5.1 responder column is the explicitly-untranscribed `respondToXidCommand` helper — both flagged to fold onto the SDL at the figc5.x backfill. New `tests/xid-negotiator.test.ts` (16 tests: per-parameter merge + the §1436 set), `tests/conformance/mdl-xid-negotiation.test.ts` (9 tests: two-station XID convergence with five concrete reverts-to merges pinned, v2.0 FRMR fallback incl. k=7, TM201/NM201 exhaustion → MDL-ERROR (C), error D/B paths), and `tests/Ax25ListenerMdl.test.ts` (the production-listener figc5.1 responder path end-to-end); the two-station harness gained per-station MDL drivers + a deferred MDL-work queue (models the async modem boundary so a just-sent XID command's MDL transition commits before its reply is handled), `startNegotiation` / `advanceTm201` drive methods, and per-station `mdlSignals`. The existing mod128-establishment suite's "every establishment frame is a SABME" assertion now filters to SABM/SABME, since a successful v2.2 connect is followed by the MDL's (legitimate) XID command.

- **XID information-field (TLV) codec** — the TS parity leg of the v2.2 arc V3 part 1 (mirrors packet.net#270). A new `src/xid.ts` module encodes/decodes the AX.25 v2.2 XID parameter-negotiation *information field* — the TLV payload carried inside an XID U-frame (§4.3.3.7, parameter table Fig 4.5, worked example Fig 4.6): `encodeXid(params)` builds FI(0x82) + GI(0x80) + GL + ordered PI/PL/PV triples; `tryParseXid(info, options?)` decodes them back, returning a `{ ok, parameters } | { ok, reason }` discriminated union (no throw on malformed input), strict-by-default with named `XidParseOptions` leniency knobs (`allowGroupLengthOverrun` / `allowTruncatedParameter`, both off by default — the `XID_PARSE_STRICT` / `XID_PARSE_LENIENT` presets) per the repo's spec-compliant-by-default philosophy; the encode path is unconditionally strict and never emits a malformed field. Covers the six AX.25 parameters: Classes of Procedures (PI=2, duplex), HDLC Optional Functions (PI=3, REJ/SREJ + modulo-8/128 + segmenter/SREJ-multiframe — the bit-fields force the spec-mandated always-1/always-0 bits), I-Field Length Rx (PI=6, in bits; `iFieldLengthRxOctets()`/`octetsToBits()` convert to/from the session's octet N1), Window Size Rx (PI=8), Ack Timer T1 (PI=9, ms), Retries N2 (PI=10). Absent parameters are modelled as `undefined` on `XidParameters` (= "use the currently-negotiated value", ¶1024) — distinct from a present-but-default value; the Tx variants PI=5/PI=7 (ISO-8885-only) and any unrecognised PI are skipped on parse. Bit-to-octet mapping is LSB-first within each octet (octet 0 first), verified independently against Fig 4.6 byte-for-byte; the codec follows the normative §4.3.3.7 table/prose over the two documented Fig 4.6 worked-example discrepancies — the Classes-of-Procedures Balanced-ABM bit sits at bit 0 (encoding 0x21, not the figure's 0x22 which mis-places it at bit 1), and the HDLC field's bytes 0x82 0xA8 0x22 literally select REJ (bit 1), not the SREJ its caption claims (we decode the bytes faithfully). New `tests/xid.test.ts` (51 tests): header/PI constants pinned to spec, an independent hand-decode of the Fig 4.6 PV bit positions, the Fig 4.6 worked example parsed field-by-field and re-encoded byte-for-byte (modulo the spliced ABM anomaly), per-parameter round-trips, absent-field handling, malformed/short-buffer rejection, and the strict-rejects/lenient-accepts pairs the repo requires for any parser leniency. This is the codec **only** — the MDL/XID negotiation FSM (figc5.1–5.8, not transcribed in `m0lte/ax25sdl`) and wiring into the session are V3 part 2, a follow-up; ax25-ts has no XID U-frame factory yet, so the codec is transport-agnostic and its output is exercised through a generic frame's encode→wire→decode round-trip.

- **Extended (mod-128) frame codec** — the TS parity leg of the v2.2 arc V1 (mirrors packet.net#266 / toward packet.net#239). `Ax25Frame` gains a `controlExtension` octet (the second control octet of an extended I/S frame — 7-bit N(R) in bits 7-1, P/F in bit 0 per Fig 4.1b; `null` for U frames and all mod-8 frames, surfaced as `isExtendedControl()`). `pollFinal()` is now mode-aware — bit 4 of the control octet under mod-8 and for U frames (1 octet in both modes), but bit 0 of the second control octet for an extended I/S frame (the common get-it-wrong bug). New mode-aware `getNs()`/`getNr()` extract 3-bit (mod-8) or 7-bit (mod-128) sequence numbers from the frame's own control width, so the dispatcher and guard bindings read them without re-deriving bits. The `iFrame`/`rr`/`rnr`/`rej`/`srej` factories take an `extended` flag (a new optional field, so existing callers are unaffected) and emit the 2-octet control field; the dispatcher passes `ctx.isExtended`. `decodeFrame` takes an `extended` parameter — an I/S control field's width isn't derivable from the octets alone, so the receive path threads the session's negotiated modulo: the `Ax25Listener` inbound pump parses at mod-8 for routing (addresses precede the control field and are modulo-independent), then re-decodes I/S frames at the matched session's modulo (`reparseAtSessionModulo`); `Ax25Stack` threads the modulo through `_handleFrame` the same way. `Establish_Data_Link` now emits SABME on an extended link (figc4.7 `mod_128` path) so the peer keeps mod-128, and the two-station conformance harness runs a full windowed mod-128 transfer (8 frames, k=8) to convergence. The classifier is unchanged — it reads only the first control octet's I/S/U + S-subtype discriminators, identical in both modulos. New `tests/frame-extended.test.ts` pins the spec octets for I and S frames, round-trips 7-bit N(S)/N(R) across the full 0–127 range, shows a wrong-modulo parse mis-frames, and regression-pins mod-8; the previously-skipped mod-128 transfer conformance test is un-skipped and green. What's still out: the driver doesn't yet *originate* mod-128 negotiation on its own (it doesn't flip `isExtended` from an inbound SABME) — that lands with V4.

- **SABME establishment + version negotiation** — the TS parity leg of the v2.2 arc V2 (mirrors packet.net#268/#269). Connect-initiation already emitted SABME vs SABM correctly (the inlined `Establish_Data_Link` branches on `ctx.isExtended`), but figc4.1/figc4.2 route the `Disconnected` DL-CONNECT path **unconditionally to `AwaitingConnection`** with no version branch (confirmed against the authoritative `DataLink_Disconnected.graphml` in `m0lte/ax25sdl` — version routing exists only on the responder SABM/SABME-received side). So a v2.2-preferred connect sent the right first frame but parked in the mod-8 state, whose T1 retry downgrades to SABM (figc4.2 `t05`) and which has no FRMR handler (so the §975 v2.0 fallback couldn't fire). Two new default-on quirks correct the figures in the runtime (off under `strictlyFaithful`): **`ax25Spec44Mod128ConnectRoutesToV22`** ([packethacking/ax25spec#44](https://github.com/packethacking/ax25spec/issues/44)) rewrites just that one transition's *target state* to `AwaitingV22Connection` (figc4.6) when `isExtended` at dispatch time — a new quirk shape (the existing #40/#42/#43 rewrite *guards*; this rewrites `next`, in `SdlSessionDriver.resolveNextState`); and **`ax25Spec45FrmrFallbackReestablishesV20`** ([packethacking/ax25spec#45](https://github.com/packethacking/ax25spec/issues/45)) forces version 2.0 (`isExtended = false`) *before* the figc4.6 t14 FRMR actions run (in `SdlSessionDriver.applyPreExecutionQuirks`), so the §975 fallback re-establishes with a SABM not a SABME (t14 draws `Establish Data Link` before `set_version_2_0`; since Establish branches on the modulo, the fallback as drawn re-sends SABME while still extended — useless against the pre-v2.2 peer that just FRMR'd). Both mirror direwolf's `ax25_link.c` (the `state_5 ? : state_1` connect routing, and `set_version_2_0` "not in flow chart" before `establish_data_link` in the FRMR handler). figc4.6 then drives the handshake: retry resends SABME, FRMR → v2.0 + re-establish (SABM → `AwaitingConnection`, mod-8), DM(F=1) → teardown, UA → Connected-extended. The driver's `STATE_PAGES` is now keyed by the figc4.6 page's real SDL name `AwaitingV22Connection` (it was mis-keyed `AwaitingConnection22` while nothing could route into it; the #44 redirect makes it reachable for the first time). The two-station harness gained a `quirks`/`buildStrictlyFaithful` knob, an `inject` (channel-bypassing event injection for the FRMR/DM rejections), per-endpoint `receivedFromPeer`/`signals` logs, and a `firedTransitions` ledger (fed by a new `onTransitionFired` driver hook — the TS analogue of the C# `Ax25Session.TransitionFired` event). New `tests/conformance/mod128-establishment.test.ts`: route-through-AwaitingV22Connection, lost-SABME-retried-as-SABME-not-SABM, FRMR→v2.0-re-establish-as-SABM (the complete mod-8 fallback), DM teardown, and a `strictlyFaithful` pin reproducing the figc4.2 defect. ax25-ts has no `Ax25SessionQuirks`-style central session record like the C# side — the quirk facility is the `Ax25SessionQuirks` *interface* + `defaultSessionQuirks`/`strictlyFaithfulSessionQuirks` presets in `src/sdl/session-quirks.ts`, which both new flags extend; the two corrections are wired directly into the driver's dispatch path.

## [0.2.0] — 2026-05-16

Adds the `Ax25Listener` class — `@packet-net/ax25` can now act as an inbound-accepting node, the single most valuable parity gap with the C# runtime. Plus two bug-fix carries from the C# runtime that were never quite right in TS either.

`@packet-net/ax25` and the `ax25sdl` companion package bump in lockstep to `0.2.0`. No breaking API changes; `Ax25Stack` and `Ax25Session` keep their existing outbound-only shape (the listener is a sibling class, not a replacement).

### Added

- **`Ax25Listener`** — first-class inbound-acceptance coordinator. Owns one `Ax25Transport`, address-filters inbound frames against `myCall`, dispatches to the per-peer `Ax25ListenerSession` (creating one on first contact — inbound SABM or outbound `listener.connect(remote)`), surfaces `sessionAccepted` + `frameTraced` events, and runs an LRU cache of cached sessions keyed by peer callsign (default cap 64). Mirrors `Packet.Ax25.Session.Ax25Listener` from the C# runtime.
- **`acceptIncoming` toggle** on `Ax25Listener` — flip to `false` to refuse all new incoming SABMs; existing sessions keep running. The listener responds with DM (figc4.1 t15) to the rejected SABM and doesn't cache the transient session.
- **Per-peer session cache** — sessions survive disconnect; the same `Ax25ListenerSession` instance is handed back on reconnect, preserving SRT/T1V smoothing and any out-of-SDL context state (e.g. `sentIFrames` between reconnect attempts).
- **Handler-exception isolation** (#140 carry-over) — every `sessionAccepted` / `frameTraced` subscriber is invoked inside a try/catch; exceptions are routed to a configurable `onHandlerError` sink (default `console.error`) and never escape the inbound pump. A buggy consumer cannot DoS the modem.
- **`Callsign`-aware `connect()`** on the listener — initiate an outbound SABM against a peer using the same per-peer cache; resolves to the session once DL-CONNECT-confirm arrives.
- **SABME factory + classifier branch** — `frame.ts` exports a `sabme(...)` factory and `classify(...)` recognises control byte 0x6F. The full mod-128 sequence-number path remains gated on the `version_2_2` predicate (still effectively false in this runtime); the addition is purely so listener tests can inject SABME and exercise figc4.1's t16 branch.

### Fixed

- **#141 carry-over: via-chain reversal on responses.** `ActionDispatcher`'s outbound frame builders now reverse the inbound trigger's digipeater chain when emitting UA / DM / RR / RNR / REJ / I / UI responses. A peer behind a two-hop digipeater chain (`[GB7CIP, MB7UR]`) sending SABM now gets UA back via the reversed chain (`[MB7UR, GB7CIP]`), per AX.25 v2.2 §C.2 Path Construction. Previously the UA had no via-chain on the wire, so the digi closest to the responder never forwarded it.
- **#143 carry-over: cache-miss DM for non-SABM frames.** When the listener receives a non-SABM frame addressed to us from a peer with no cached session (DISC / RR / RNR / REJ / SREJ / I / FRMR / XID / TEST), it now builds a transient Disconnected session, dispatches the event so the SDL's per-event arm fires (DISC → t13 → DM; everything else → t05 all_other_commands → DM), then drops the transient session without caching it. Previously these frames were silently dropped at the cache-miss filter, leaving the peer's half-open session view hanging.

### Test count

- Unit tests +30 across `Ax25Listener.test.ts` (5), `Ax25ListenerConcurrency.test.ts` (7), `Ax25ListenerMultiPeer.test.ts` (7), `Ax25ListenerRejectAndEdge.test.ts` (10), and `ActionDispatcherViaChain.test.ts` (1) — 1:1 mirror of the 30 C# listener tests.
- Integration tests +3 across `listener-netsim-multi-peer.test.ts` (2) and `listener-linbpq-initiates.test.ts` (1).

### Known limitations

- Per-peer cache is single-threaded (JS is); the C# listener uses a `ConcurrentDictionary` for the same role. The TS surface is event-driven on a single event loop, so the C# concurrency primitives are unnecessary.
- The listener-owned session class is exported as `Ax25ListenerSession` to avoid name-collision with the existing `Ax25Session` from `Ax25Stack`. Functionally similar (both wrap an `SdlSessionDriver`) but the API surfaces differ — `Ax25Session` exposes outbound-flavoured `connect` / `write` / `disconnect`; `Ax25ListenerSession` exposes raw `postEvent` + signal subscription.

### Known limitations

- **k=1 single-outstanding-I-frame window** — throughput will be SRT-bound on real links.
- **REJ/SREJ recovery loops** — wire frames emit, but the recovery subroutines are no-op stubs (figc4.7 paths not yet walked).

- **No mod-128 (SABME)** — the SDL `version_2_2` predicate returns false.
- **`via` digipeater paths** — frame factories and codec round-trip them, but `stack.connect({ via: [...] })` throws.
- **No AGW / AXUDP / audio transports** — Web Serial + TCP only.
- **No inbound connection acceptance** — SABM addressed to us with no matching outbound session is silently dropped.
